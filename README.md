# Описание

Микросервис отслеживания подписок пользователей. Не управляет пользователями.

# Технологии

Использовались:
 - postgresql
 - goose для миграций
 - sqlc для генерации go кода из sql
 - net/http оборачивалось для использования middlewares и обработки ошибок
 - slog для логирования
 - включен swagger + документация
 - docker compose

# Запуск

Скопируйте `.env.example` в `.env`.

Запуск в docker:
```
docker compose up -d --build
```

Остановка в docker:
```
docker compose down
```

Логи приложения:
```
docker compose logs app
```

Путь до main - `cmd/app/main.go`.

# Эндпоинты

Создать подписку:
```
POST /subs/
```

Получить подписку:
```
GET /subs/{id}
```

Редактировать подписку:
```
PUT /subs/{id}
```

Удалить подписку:
```
DELETE /subs/{id}
```

Вывести подписки по параметрам:
```
GET /subs/list
```

Посчет стоимости подписок за период:
```
GET /subs/total
```

Подробнее, а так же параметры, тела запросов и тела ответов смотреть в документации swagger (`cmd/app/docs/`).

Чтобы попасть в swagger, нужно перейти в
```
/swagger/index.html
```

# Архитектурные решения

Переменные окружения содержатся в `.env`. Config их парсит и они становятся доступными для использования. Смотреть `internal/config`.

Все запросы к БД написаны на чистом sql и внедрены в код с помощью sqlc. Все миграции применяются при старте приложения. sqlc настроен так, чтобы генерировать интерфейс, тем самым получения данных можно мокать и тестировать другие слои приложения. Смотреть `db/sqlc`, `internal/store`.

В проекте запросы регистрируются через Application. Это структура, которая оборачивает net/http и позволяет внедрять зависимости в обработчики запросов. Кроме того, позволяет добавлять middlewares, содержит в себе общие функции для сетевого слоя. Смотреть `internal/application`.

Middleware это то, что вызывается перед обработчиком запроса. Это переиспользуемые функции, позволяющие сделать общие вещи и разгрузить обработчик запроса. Смотреть `internal/middlewares`.

Обработчики запросов имеют доступ к нужным зависимостям и могут их использовать. Они отправляют успешный ответ самостоятельно, а ошибки возвращают. Ошибки по цепочке ловит error middleware и правильно отвечает пользователю. Смотреть `internal/handlers`.

Все используемые представления находятся в `internal/dto`. Общие функции в `internal/utils`.
